Git



Co to jest Git i w czym ma nam pomóc ?

asd

asd

asd

Git jest to narzędzie do kontroli wersji, dzięki któremu możemy zarządzać wersjami plików naszego projektu. Wszyskie projekty korzystające z Gita mają folder .git znajdujący się w folderze projektu na najwyższym poziomie. Z jego największych funkcjonalności można wyróżnić:

  • Historia pliku / projektu - dzięki gitowi możemy, np. po dwóch latach rozwijania projektu podejrzeć wersję danego pliku od początku istnienia projektu do teraźniejszej wersji
  • Kolaboracja - dzięki kombinacji np. Git + GitHub możemy pracować w zespołach, gdzie każdy pracuje nad swoją częścią projektu oraz pozwala na rozwiązywanie ewentualnych konfliktów jeśli wasza zmiana pokrywa się ze zmianą innego członka zespołu
  • Praca na branch-ach (gałęziach) - jest to sposób na wydzielenie zmian, dzięki czemu możemy pracować nad dwoma niezależnymi zmianami jednocześnie więcej o branch-ach potem
  • Własna wersja projektu - standardowo projekty trzyma się na zdalnych repozytoriach (np. GitHub), ale Git pozwala na probranie takiego repozytorium na nasz system, co pozwala na zmiany w projekcie, które nie mają wpływu na osoby z zespołu. Po skończeniu wprowadzaniu naszych zmian możemy je “wypchnąć” na zdalne repozytorium, żeby członkowie zepołu widzieli wasze zmiany.

Czym jest GitHub ?

GitHub jest to zdalne repozytorium wykorzystujące Gita. Dzięki niemu możemy trzymać nasze repozytoria zdalnie, przez co możemy dzielić się naszym kodem z innymi oraz prowadzić razem projekt. Oprócz GitHuba jest wiele innych serwisów podobnego zastosowania jak na przykład GitLab, Bitbucket, SourceForge, itd. Jednak GitHub jest najpopularniejszy i dość prosty w użyciu.

W jaki sposób działa Git

Nie będziemy wchodzić w szczegóły jak działa Git, ale postaram się wam zobrazować jak komendy Gita wpływają na to jak postrzega wasze pliki projektu, do tego posłużę się diagramem.

sequenceDiagram participant U as Unstaged participant S as Staged participant C as Commited participant P as Pushed U ->> S: git add . S ->> C: git commit -m "Mój pierwszy Commit" C ->> P: git push
Unstaged
Plik istnieje u nas w plikach projektu, ale nie jest brany pod uwagę przez Gita
Staged
Plik został dodany do kontroli wersji Gita, ale jeszcze nie ma zatwierdzonych zmian
Commited
Po zatwierdzeniu zmian plik jest pełnoprawną częścią projektu
Pushed
Zmiany, których dokonaliśmy lokalnie zostały wypchnięte na zdalne repozytorium (np. GitHub) i mogę zrobić pull requesta, bądź koledzy mogą podejrzeć jakie zmiany wprowadzam do projektu

Jak widzimy na diagramie praca z Gitem dzieli się na parę części. W stanie unstaged znajdują się pliki, które nie podlegają jeszcze pod wersjonowanie Gita. Jeśli chcemy żeby Git brał dany plik pod uwagę musimy najpierw go dodać wołając git add ., a potem ‘zacommitować’, czyli potwierdzić zmianą i nadać jej nazwę. Jeden commit może zawierać klika zmian, np. dodanie nowego pliku, modyfikacja istniejącego pliku, przeniesienie pliku z jednego miejsca w drugie i inne modyfikacje na plikach. Commit wykonujemy za pomocą git commit -m "Message commit-u". Kiedy nasza zmiana jest gotowa wypychamy nasze commity na zdalne repozytorium za pomocą git push, żeby nasze zmiany byłby widoczne na GitHubie i inni mogli je zobaczyć. Jest to bardzo uproszczony model pracy z Gitem i w rzeczywistości będziecie musieli wykonać więcej komend, które poznacie dalej.

Branch-e i Git flow

Git pozwala przez platformy takie jak GitHub pracować w zespołach nad jednym projektem. Git flow jest to jedno z podejść jakie jest wykorzystywane podczas pracy zespołowej nad projektem (U mnie w pracy cały czas wykorzystywany). Wykorzystuje on feature branch-e oraz “główne” branch-e. Głównymi branch-ami są master/main i developlment/dev.

Master branch
Jest to branch, na którym znajduje się kod produkcyjny, czyli taki który jest ostateczną wersją naszego programu
Development branch
Na tym branch-u znajduje się wersja naszego programu z nowymi funkcjonalnościami, gdy posiada już wystarczającą ilość zmian, które wprowadzają ważną funkcjonalność wtedy development branch przechodzi testy i jest wykonywany merge z master branch-em, żeby wypuścić nowe funkcjonalności dla użytkowników aplikacji
Feature branch i merge
Jest to branch, którego tworzymy na początku wprowadzania nowej funkcjonalności, np. chcemy zmienić to jak wygląda główna strona naszych notatek. Odbijamy się wtedy od branch-a development (czyli tworzymy branch-a, który będzie się zaczynał od najnowszej wersji branch-a development), wprowadzamy nasze zmiany i na koniec tworzymy pr-a (pull request-a) z naszego branch-a do branch-a development. Po review, zamykamy pull request-a kończąc go akcją merge i wprowadzamy nowe zmiany na development, a feature branch-a usuwamy. Merge nazywamy akcję połączenia dwóch branch-y. Na diagramie wykonane są 2 akcje merge, pierwsza dorzucająca funkcjonalności do develop-a zmiany z feature/add-new-style-to-notes-page, a potem druga dorzucająca zmiany do main-a z develop-a.

Tutaj diagram obrazujący jak możecie sobie wyobrażać git flow i jak wygląda praca w zespole. Jak widzicie na diagramie każdy commit ma swój unikalny identyfikator UUID, a wersje programu (czyli każdy commit na main-ie) zazwyczaj jest otagowany wersją, w tym przypadku mamy wersję v0.0.1, a po zmianach v0.0.2.

Pull request-y i review

Pull request-y i review odnoszą się trochę do metodyki Git flow. Mają one na celu sprawdzenia przez innych członków zespołu naszej zmiany, którą chcemy wprowadzić do projektu. Pull request-em nazywamy nasze żądanie o sprawdzenie kodu w celu dodania go do zazwyczaj branch-a development, a review nazywamy proces oceniania kodu w danym pull request-cie.

Typowe operacje na Git-cie

Następne sekcje pokryją jedne z ważniejszych operacji, z którymi będziecie się spotykać na co dzień. Każdą poniższą operację można zrobić na X sposobów, ale postaram się przedstawić wam te najprostsze.

Tworzenie nowego projektu

Najłatwiej jest zrobić nowe zdalne repozytorium na GitHubie i zrobić git clone <url-zdalnego-repo.git>

git clone https://github.com/JanCieslak/notki.git

Tworzenie nowego branch-a

git checkout -b <nazwa-branch-a>

Commit-owanie nowych zmian

# Żeby zobaczyć jakie pliki mogą być dodane możemy zawołać (ponieważ były zmienione, dodane, zmodyfikowane, itd.)
git status

# Weź wszystko co się zmieniło od ostatniego commit-a
git add .

# Można też dodać pojedyncze pliki
git add plik1.txt plik2.txt

# Zatwierdź te zmiany
git commit -m "Dodanie nowej funkcjonalności"

Wypychanie zmian na zdalne repozytorium

git push

# Wylistuj remote-y
git remote

# Lub gdy nie ustaliliśmy połączenia lokalnego branch-a ze zdalnym
git push -u <nazwa-remote> <nazwa-zdalnego-branch-a>

# Gdy jesteśmy np na branch-u development, a nasz remote nazywa się origin to zrobimy
git push -u origin development

Pliki specjalne

Istnieje kilka plików bądź folderów specjalnych dla Gita, oto pare z nich.

.git

Jest to folder mówiący nam, że dany folder jest projektem używającym Gita, raczej nie powinniśmy nic z nim robić, ponieważ jest on tylko do użytku przez Gita. Przez różne modyfikacje tego pliku Git mógłby nam działać niepoprawnie.

.gitignore

Jest to plik zawierający pliku lub foldery, których nie chcemy żeby Git wersjonował. Często wybiera się pliki lub foldery programów, których używamy. Na przykład VSCode sam czasem stworzy folder w naszym projekcie i nie chcemy że był on dodany do publicznego repozytorium, ponieważ każdy VSCode jest skonfigurowany inaczej i taki plik może wyglądać zupełnie inaczej u innej osoby. Tutaj znajdziecie generator takiego pliku, przez wprowadzenie rzeczy których używamy on wygeneruje nam plik, który powinien pokryć rzeczy których nie powinniśmy wrzucać na GitHuba.

Komendy

# Zacznij używać kontroli wersji Git w danym folderze
git init

# Pobierz zdalne repozytorium
git clone <remote.git>

# Sprawdź status plików w lokalnym repozytorium
git status

# Sprawdź ostatnie commit-y w wersji skróconej
git log --oneline

# Dodaj pliki do paczki zmian (commit-a)
git add <files>

# Dodaj wszystkie pliki zmienione do paczki zmian (commit-a)
git add .

# Zatwierdź paczkę zmian nadając jej nazwę
git commit -m "Mój pierwszy commit"

# Wylistuj remote-y (zdalne repozytoria)
git remote

# Dodaj zdalne repozytorium
git remote add origin <nazwa-repo.git>

# Wypchnij wszystkie swoje commit-y do zdalnego repozytorium, jeśli robimy to pierwszy branch to musimy
# podać nazwę remote oraz nazwę branch-a, który będzie zrobiony w zdalnym repozytorium (zazwyczaj podajemy tą samą nazwę
# branch-a jaką mamy lokalną).
git push -u <remote> <remote-branch>

# Gdy już ustawimy powyższą komendą, kolejne push-e można wykonywać komendą
git push

# Wylistuj wszystkie lokalne branch-e
git branch

# Stwórz nowego branch-a odbijając się z aktualnego branch-a na którym jesteśmy i od razu przełącz się na niego
git checkout -b <branch-name>

# Przełącz się na podany branch
git checkout <branch-name>

# Pobierz wszystkie zmiany ze zdalnego repozytorium nie zmieniając plików lokalnych
git fetch

# Pobiera zmiany ze zdalnego repozytorium i aktualizuje branch-a na którym aktualnie jesteśmy
git pull

# Zrób rebase na branchu bazowym z brancha ze zmianami
git rebase <branch-bazowy> <branch-ze-zmianami>

# Łączy zmiany z podanego branch'a do aktualnego branch'a
git merge <branch-name>

Słownik

Git
Narzędzie do kontroli wersji plików
GitHub
Serwis przeznaczony do projektów programistycznych wykorzystujący Gita jako system kontroli wersji
branch
Jeden z gałęzi w drzewie Gita. Każdy branch zazwyczaj posiada swoje zmiany, które koniec końców i tak powinny trafić na branch development a później master
commit
Paczka zmian, którym nadajemy nazwę
pull request
Poinformowanie osób zaangażowanych w projekt o nowych zmianach z prośbą o zrobienie review i ewentualnym zaakceptowaniu zmian jeśli spełniają one oczekiwania osoby robiącej review
review
Proces ocenienia pull request-a, zazwyczaj przez inne osoby z zespołu
